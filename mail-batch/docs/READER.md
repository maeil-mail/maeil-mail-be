## Paging Reader 선택 이유


### Cursor는 별로인듯
CursorItemReader 기본 내용 :

- 커서 기반 처리 방식은 데이터베이스와 끊김 없는 연결을 유지하면서, 데이터를 ResultSet을 통해 순차적으로 가져오는 방식
- CursorItemReader가 초기화(open)될 때 빌더를 통해 지정된 SQL 쿼리를 실행하고, 그 결과를 가리키는 커서를 생성한다. 이후 read()가 호출될 때마다 ResultSet.next()를 실행하며
  한 행씩 데이터를 가져온다.
- 매번 새로운 쿼리를 실행하는 것이 아닌 이미 열린 ResultSet에서 커서를 이동하면서 데이터를 읽어오는 구조
- 메모리 사용량을 최소화할 수 있지만, 데이터베이스와의 연결을 유지한 채 진행되므로, 긴 배치 작업 동안 커넥션이 너무 오래 유지된다.
- ResultSet.next()를 호출할때마다 한행씩 가져오는 것이 아니라 JDBC 드라이버가 여러 row를 가져와서 ResultSet 내부 버퍼에 저장한다.
- next()는 호출할때마다 버퍼에서 데이터를 가져온다.
- MySQL 드라이버는 useCursorFetch connection property 설정이 없으면 기본적으로 쿼리 결과의 모든 행을 한 번에 메모리로 가져온다
- Step 트랜잭션과 별도의 데이터베이스 커넥션을 사용하기 때문에 Step 트랜잭션의 영향을 받지 않는다.
- Step 트랜잭션이 실패하면 jumpToItem()을 호출하여 이전 실패 지점에서 다시 시작하게 되는데, 이때 ResultSet.next()를 호출해서 실패 지점까지 커서를 이동시키는데, 만약 쿼리 결과의 순서가
  매번 다르다면 ResultSet.next()로 이동한 지점이 이전 실행의 그 지점이라고 보장할 수 없다.

메모 :

- MySQL에서 cursorItemReader를 사용하게 되면, reader에서 DB와의 별도의 커넥션을 사용하여 데이터를 스트리밍한다.
- Step 자체에서 데이터를 수정해도 Reader에서 항상 같은 데이터가 보여지는 이유는 MVCC와 관련되어 있는 것으로 생각된다.
    - 대량의 데이터에 대한 백업 데이터를 가지고 있어야, 청크 트랜잭션의 결과가 Reader 트랜잭션 조회 결과에 영향을 미치지 않기 때문이다.
    - MySQL에서는 언두 로그(트랜잭션과 격리 수준을 보장하기 위해 백업해둔 변경 전의 데이터)를 관리한다.

언두 로그에 대한 추가 내용 :

- 언두 로그의 역할은 트랜잭션 보장, 격리 수준 보장이다.
- 커밋 여부와 무관하게 실제 데이터와 버퍼풀의 내용을 변경하고, 언두 영역에 변경 전의 값을 백업한다.
    - 커밋되면 현상태를 유지하고, 롤백되면 백업 데이터로 북구한다.
- 따라서, 다음과 같은 상황에서는 좋지 않다.
    - 대량의 데이터를 변경, 삭제하는 경우
    - 트랜잭션을 오래 유지하는 경우

결론 :

- 배치 작업 동안 데이터베이스 커넥션이 오래 지속되는 경우, 3가지 문제가 발생한다.
    - `커넥션이 종료되는 상황`
        - 커서를 사용하게 된다면, 실시간으로 변경되는 데이터셋의 스냅샷을 사용하여 안정적인 처리가 가능해진다.
        - 다만, Reader 커넥션이 종료된다면 다시 데이터셋을 조회하기 때문에 위 이점이 흐려진다.
    - `백업 데이터 공간 증가: 실측이 필요하나, 위 문제가 참이라면 이 리스크를 짊어질 필요가 없다.`
        - API 애플리케이션에서 subscribe 데이터셋에 대한 수정을 실시간으로 하고 있는 상황에서 배치 애플리케이션에서 해당 데이터셋에 대한 트랜잭션을 사용하고 있다면,
        - subscribe 데이터셋에 대한 수정 트랜잭션으로 생성된 언두 로그(리두는 조사안해봄)는 배치 애플리케이션의 트랜잭션이 끝난 이후에 삭제된다.
        - 조회 성능 문제 발생 가능성 있음
            - 언두 로그의 공간 사용량이 증가하면, 단순 레코드 조회 쿼리를 처리하는 시간이 길어진다. -> 조회를 위해 언두 로그의 이력을 어느 정도 파악해야하기 때문
            - 장기간 유지되는 활성 상태 트랜잭션은 성능상 좋지 않음. -> 모니터링 필요성
    - `리소스 고갈: 크게 문제가 되지 않을 것이라고 판단.`
        - 배치 애플리케이션 관점에서 하나의 커넥션을 오래 사용한다고, 배치 앱의 다른 부분에서 사용할 커넥션이 부족해지지 않는다.
            - API, 배치 각각의 커넥션 풀을 사용하기 때문이다.
    - `ThreadSafe하지 않음: 현 단계에서 고려 요소가 아님`

### JpaPagingItemReader는 사용하기 어려움

기본 내용 :

- limit, offset 기반 페이징 쿼리를 사용한다.

limit, offset 한계 :
- DB는 매 페이지 요청마다 백만 개의 데이터를 메모리에 올리고 정렬하는 작업을 반복해야 한다. 
- 뒤쪽 페이지로 갈수록 스킵해야 하는 레코드 수가 늘어난다.
- 전체 데이터를 매번 메모리에 올리고 정렬하는 비용도 발생하고, 필요한 레코드에 도달하기까지 버려야 하는 데이터도 점점 늘어남
- 실시간 데이터셋 처리에 취약함
    - 페이지 1을 읽고, 새로운 데이터가 맨 앞에 삽입되면 페이지 2를 읽을 때, 페이지 1의 마지막 레코드 중복 조회됨
    - 페이지 1을 읽고, 앞쪽 데이터가 삭제되면 페이지 2를 읽을 때, 원래 페이지 2에 있어야 할 일부 레코드가 누락됨
- 데이터베이스 메모리도 많이 차지한다. DB는 전체 결과를 메모리에 로딩한 후 필요한 부분만 잘라서 반환하기 때문

주의 사항 :
- Hibernate는 Fetch Join과 페이징(LIMIT/OFFSET)을 함께 사용하면 쿼리에서 페이징을 적용하지 않고 전체 데이터를 메모리에 로드한 후 애플리케이션에서 페이징을 수행한다.
